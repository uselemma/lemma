This page covers the trace-level context you set when you open a trace: thread IDs that group multi-turn conversations, user IDs that attach an end user, and the fields Lemma normalizes each one to.

## Threads

When one user has a back-and-forth with your agent, each turn is its own trace. Give related turns the same `threadId` so Lemma groups them into one conversation.

<Tabs>
  <Tab title="TypeScript">
    ```typescript
    await lemma.trace(
      {
        name: "support-agent",
        input: userMessage,
        threadId: conversationId,
      },
      async (trace) => {
        return runAgent(userMessage, trace);
      },
    );
    ```
  </Tab>
  <Tab title="Python">
    ```python
    async def run(trace):
        return await run_agent(user_message, trace)

    await lemma.async_trace(
        "support-agent",
        run,
        input=user_message,
        thread_id=conversation_id,
    )
    ```
  </Tab>
</Tabs>

## Users

Set `userId` to attach an end user to the trace, so you can filter and group traces by who triggered them.

<Tabs>
  <Tab title="TypeScript">
    ```typescript
    await lemma.trace(
      {
        name: "support-agent",
        input: userMessage,
        threadId: conversationId,
        userId: user.id,
      },
      async (trace) => {
        return runAgent(userMessage, trace);
      },
    );
    ```
  </Tab>
  <Tab title="Python">
    ```python
    async def run(trace):
        return await run_agent(user_message, trace)

    await lemma.async_trace(
        "support-agent",
        run,
        input=user_message,
        thread_id=conversation_id,
        user_id=user.id,
    )
    ```
  </Tab>
</Tabs>

## Environment and service name

To separate environments or set a service name, use the mechanism that matches your goal:

| Goal | Mechanism |
| --- | --- |
| Isolate staging from production | Use separate Lemma projects |
| Label the environment on a trace | Set `metadata["deployment.environment.name"]` |
| Set a custom service name | Use the OpenTelemetry Protocol (OTLP) resource `service.name`; the SDK ingest path always stores `service_name: "lemma-sdk"` |

Attach the environment label through trace `metadata`:

<Tabs>
  <Tab title="TypeScript">
    ```typescript
    await lemma.trace(
      {
        name: "support-agent",
        input: userMessage,
        metadata: { "deployment.environment.name": "staging" },
      },
      async (trace) => runAgent(userMessage, trace),
    );
    ```
  </Tab>
  <Tab title="Python">
    ```python
    lemma.trace(
        "support-agent",
        lambda trace: run_agent(user_message, trace),
        input=user_message,
        metadata={"deployment.environment.name": "staging"},
    )
    ```
  </Tab>
</Tabs>

<Note>
  `TraceOptions.environment` was removed and was never persisted on the SDK ingest path. Use a separate project or the `deployment.environment.name` metadata key instead.
</Note>

## Context field reference

Each context field maps to an SDK option, a client field, and the attribute ingest normalizes it to:

| Context | SDK option | Client field | Ingest normalizes to |
| --- | --- | --- | --- |
| Agent name | `name` | `trace.name` | `gen_ai.agent.name`, `ai.agent.name` |
| Thread / conversation | `threadId` / `thread_id` | `trace.thread_id` | `lemma.thread_id` |
| User | `userId` / `user_id` | `trace.user_id` | `user.id`, `enduser.id` |
| Environment | `metadata["deployment.environment.name"]` | `trace.metadata` | `deployment.environment.name` |

See the [trace contract](/reference/trace-contract) for how each field is sent and normalized.
